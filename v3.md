Entendido: lo quieres **rudo**, **a prueba de errores**, y que aunque el usuario (o la red, o la IA) haga tonterías, **no se rompa** ni corrompa datos.

Aquí tienes un plan de “hardening” **completo** (frontend + backend/edge + DB + Docker + pruebas + operación). Si haces esto, el proyecto se vuelve *industrial*.

---

# 1) Principios no negociables (si no, siempre habrá caos)

## 1.1 Una sola verdad por capa

* **Una sola fuente de envs** (validadas) y **un solo supabase client** por contexto (browser/server/middleware).
* **Un solo AI gateway** (nada de mock mezclado con real sin flag).
* **Un solo modo backend real** (Supabase completo, no PostgREST pelón si usas Storage/Auth/Functions).

## 1.2 La DB es la policía

No confíes en el frontend. Todo dato importante debe quedar protegido por:

* constraints (`NOT NULL`, `CHECK`, `FK`, `UNIQUE`)
* RLS
* triggers (ej. `user_id = auth.uid()`)

---

# 2) Hardening del FRONTEND (UX + validación + “no se puede romper”)

## 2.1 Validación fuerte antes de enviar

* Zod + react-hook-form en TODOS los formularios:

  * Item: `name`, `quantity>=1`, `photo` requerido, `belongs_to` requerido si accessory
  * Container: `label` requerido, `location_id` requerido

✅ Resultado: reduces 70% errores antes de tocar backend.

## 2.2 Estados de UI obligatorios (sin “pantalla muerta”)

Cada pantalla debe tener:

* loading skeleton
* empty state (sin datos)
* error state con acción “reintentar”
* success toast

## 2.3 Anti-doble submit

* Botones se deshabilitan mientras `mutation.isPending`
* Usa “idempotency key” en requests importantes (ver sección backend)

## 2.4 Manejo de red mala / offline

* Detectar offline (navigator.onLine) y mostrar banner “sin conexión”.
* No dejar “subir foto” si está offline (o cola offline si quieres pro).

## 2.5 Preprocesamiento de imagen en cliente (clave)

Antes de subir:

* limitar tamaño (ej 1600px max)
* comprimir (calidad 0.75)
* convertir a jpg/webp

✅ Evita:

* requests gigantes
* timeouts
* costos y fallos en IA

---

# 3) Hardening del BACKEND (Edge Functions) — lo más crítico

## 3.1 Contratos estrictos (Zod server-side)

Para cada Edge Function:

* valida request (`photo_path`, `mime_type`)
* valida response de IA (JSON schema)
* si no cumple → retry / fallback / error controlado

✅ “La IA se equivoca” y aun así tu app no se rompe.

## 3.2 Timeouts + retries + backoff + circuit breaker

Obligatorio en IA:

* timeout (10–15s)
* retry x3 con backoff exponencial SOLO en 429/5xx
* fallback Gemini→Groq si falla por:

  * 401/403 (key mala)
  * 429 (rate limit)
  * 5xx
  * timeout
  * JSON inválido

**Circuit breaker**:

* si Gemini falla 5 veces seguidas en 2 min → no lo intentes por 2 min, usa Groq directo.

✅ Evitas “cascadas” de error.

## 3.3 Rate limiting (para que no te spameen IA)

En `analyze-item`:

* máximo X requests/min por user_id
* si excede → 429 “espera”

Puedes hacerlo con:

* tabla `ai_rate_limits` (simple)
* o Redis si lo tienes

## 3.4 Seguridad dura (sin atajos)

En cada function:

* exige JWT Bearer
* `photo_path.startsWith(user.id)`
* nunca uses service role para cosas del usuario (solo cuando sea estrictamente necesario)

---

# 4) Hardening de DATABASE (integridad + consistencia)

## 4.1 `user_id` NO debe venir del cliente

Haz que la DB lo asigne:

* `user_id uuid not null default auth.uid()`

Y en policies:

* `with check (user_id = auth.uid())`

✅ Nunca entra un registro con user_id equivocado.

## 4.2 Constraints que evitan basura

En `items`:

* `quantity >= 1`
* `name` no vacío
* `item_type='accessory' => belongs_to_item_id NOT NULL`
* `item_type='device' => belongs_to_item_id IS NULL`

En `containers`:

* `label` no vacío
* `location_id` siempre válido del mismo usuario (FK compuesta ya la tienes)

## 4.3 Soft delete + auditoría (pro)

* `deleted_at` en items/containers (en vez de borrar)
* tabla `audit_log` (qué cambió, cuándo, user_id)

✅ Recuperas errores humanos.

## 4.4 Índices para escala real

* `items(user_id, container_id)`
* `items(user_id, item_type)`
* `items(search_tsv) GIN`
* `items(embedding) ivfflat` (cuando haya volumen)
* `attributes(user_id, item_id)`
* `containers(user_id, location_id)`

---

# 5) Hardening de STORAGE (fotos que no “se pierden”)

## 5.1 Reglas de bucket privadas

* bucket `item-photos` privado
* policies owner-only (ya lo contemplaste)

## 5.2 Rutas ordenadas e inmutables

`<user_id>/<item_id>/<timestamp>.jpg`

✅ Nunca se pisa una foto por accidente.

## 5.3 Thumbnails (rendimiento)

* Generar miniaturas (imgproxy o function)
* UI usa miniatura en grids, full size solo en detalle

---

# 6) Hardening de DOCKER / ejecución local (sin sorpresas)

## 6.1 Healthchecks

* `web` healthcheck (GET /)
* `kong` healthcheck
* `db` healthcheck

## 6.2 Resource limits

* memoria/cpu para evitar que IA o web se coma todo

## 6.3 Variables internas vs externas (la causa #1 de errores)

* `NEXT_PUBLIC_SUPABASE_URL` = host (browser)
* `SUPABASE_INTERNAL_URL` = docker network (middleware SSR)

✅ Sin esto, siempre tendrás errores raros.

---

# 7) Pruebas: “a prueba de errores” = pruebas reales

## 7.1 Plan mínimo de pruebas (obligatorio)

### Unit (rápidas)

* validaciones Zod
* helpers de storage (path builder)
* parsing JSON IA

### Integration (DB real)

* insertar container/item con RLS (usuario A no ve usuario B)

### E2E (Playwright)

1. login → dashboard
2. crear ubicación → crear caja
3. agregar item con foto
4. IA autocompleta
5. buscar texto y semántico

✅ Esto evita regresiones.

## 7.2 “Chaos tests” (los que hacen el proyecto rudo)

* IA devuelve JSON inválido → debe reintentar y/o fallback
* IA tarda 20s → timeout → fallback
* imagen > 4MB → frontend la reduce; backend rechaza si llega
* se cae Supabase storage → UI muestra error y reintentar
* request duplicada → idempotencia evita duplicados

---

# 8) Operación: backups y recuperación (sin esto no es “rudo”)

## 8.1 Backups automáticos

* `pg_dump` diario con rotación (7/30 días)
* export de storage (o sync)

## 8.2 Restore probado

* 1 vez: simula pérdida, restaura en local y confirma

✅ Si no puedes restaurar, no tienes backups.

---

# 9) Checklist final “RUDO” (Definition of Done)

La app es “a prueba de errores” cuando:

* [ ] Ningún insert/update depende de `user_id` del cliente
* [ ] RLS y Storage policies bloquean accesos cruzados
* [ ] Forms validan todo (Zod) y no permiten submits inválidos
* [ ] IA tiene timeout + retry + fallback + JSON validation
* [ ] Rate limiting IA activo
* [ ] Imágenes se comprimen antes de subir + límite servidor
* [ ] Errores siempre muestran UI útil (reintentar / fallback manual)
* [ ] E2E corre y pasa en CI
* [ ] Backups automáticos + restore probado
* [ ] Healthchecks + logs estructurados
